{{existing_data}}

This is the latest email/s from the client:
{{latest_email}}
Only extract deliverables that are explicitly mentioned or clearly implied in the email. Do not estimate or fabricate deliverables from emails that lack actionable project scope (e.g. welcome emails, introductions, general check-ins).
If data exists, then only provide a response with changes to the current saved information.

Write down your thinking before you write the data under a [[RESULT]] heading.
First, think through what the existing deliverables are, when they are due, and who they are assigned to. What is their status? Have they already been submitted? 
Then, think through what the email's intent is, whether there are any explicit or hinted changes to the deliverables. Remember a deliverable can't be approved if it was not submitted for review.
Finally, using your analysis, understand whether you should update any deliverables. 
A. If there's nothing to update, do not provide JSON data.
B. If there are updates then provide a complete and up to date overview in the JSON format under a [[RESULT]] heading. 

Your JSON data must have the following structure, in the following schema:

Dict containing a single "deliverables" key, the value of which is a list.
- the list contains dict[str,str] elements, each dict contains these keys:
-- "title" : str value, a very concise summary of the deliverable
-- "spec" : str value, a detailed description and when possible technical, of what is needed 
-- "due_on" : str value. Describes when the deliverable should be submitted and contain only time information.
-- "is_submitted" : bool value containing True or False. If the deliverable was sent to client
-- "is_approved" : bool value containing True or False. if the deliverable was confirmed in writing as finished and satisfactory by the client.
-- "assigned_to" : str value in the format Firstname Lastname <email address>
-- "status" : str value. a very concise status summary
